WHAT IS TALOS?
--------------
Talos is an AI-powered HVAC hiring platform. Employers post HVAC jobs, candidates
apply and upload resumes, and the system uses Claude (Anthropic AI) to score/rank
candidates, generate job descriptions, and assist with pipeline communication.

Domain: gotalos.io
GitHub: https://github.com/jdrissel5253-netizen/Talos.git
AWS Account: talos-deployer IAM user, us-east-1 region

FRONTEND

BACKEND
-------
  Location:   backend-node/
  Framework:  Node.js + Express
  Database:   PostgreSQL (AWS RDS in production, local pg in dev)
  Auth:       JWT (jsonwebtoken), bcrypt for passwords
  File store: AWS S3 (resumes uploaded by candidates)
  AI:         Anthropic Claude API (resume analysis, job description generation)
  Email:      AWS SES (via nodemailer) for candidate outreach
  OAuth:      Google OAuth2 for Gmail integration
  Hosting:    AWS Elastic Beanstalk (environment: Talos-backend-prod,
              app name: talos-backend-v2)

DATABASE (PostgreSQL)
---------------------
  Key tables:
    jobs            — job postings (title, description, requirements, pay, etc.)
    candidates      — applicants (resume S3 key, score, tier, status, email)
    pipeline        — candidate pipeline status per job
    communication_log — outbound emails/messages to candidates
    audit_log       — immutable audit trail for status changes, job edits/deletes
    users           — employer accounts (email, hashed password)
    gmail_tokens    — stored OAuth tokens for Gmail integration
    migrations      — schema migration tracking

  Schema notes:
    - salary_max >= salary_min enforced by CHECK constraint (NUMERIC(10,2))
    - pay_range_max >= pay_range_min enforced by CHECK constraint
    - applicant_email column on candidates (24-hour duplicate application block)
    - Pipeline status changes enforced by a state machine (PIPELINE_TRANSITIONS)

================================================================================
  PIPELINE STATUS STATE MACHINE
================================================================================

  Valid statuses: new → reviewed → approved → contacted → backup → rejected

  Allowed transitions (PIPELINE_TRANSITIONS in databaseService.js):
    new       → reviewed, approved, backup, rejected
    reviewed  → approved, backup, rejected
    approved  → contacted, backup, rejected
    contacted → backup, rejected
    backup    → approved, contacted, rejected
    rejected  → backup

  Enforced server-side on both single and bulk status updates.

  SECURITY WORK COMPLETED (all 10 audit sections)

SECTION 5 — Input Validation
  - sanitize helper: trimString, email, phone, enumValue, positiveInt, nonNegativeNumber
  - All routes validate and sanitize inputs before DB queries
  - Auth: email format check, password min 8 chars
  - Apply: email/phone format, max lengths, per-field errors
  - Jobs: title required, all strings trimmed + max lengths, numeric validation
  - Pipeline: enum validation on status/tier/communicationType
  - Anthropic API: 3-attempt exponential backoff (1s/2s/4s)
  - Frontend: on-blur validation, inline errors, response.ok checks everywhere

SECTION 6 — Data Integrity
  - Pipeline state machine enforced server-side (no invalid transitions)
  - Salary/pay fields: NUMERIC(10,2) with CHECK constraints in DB
  - Pay range min <= max validated in job routes
  - Duplicate application block: 24-hour window per email+job
  - audit_log table: immutable record of status changes, job edits/deletes
  - scoringService.js: extracted shared scoring logic (no duplication)
  - processResumeInBackground: DB ops wrapped in PostgreSQL transaction
  - Email: log-before-send pattern (pending → sent/failed)

SECTION 7 — Logging & Observability
  - logger.js: levels (debug/info/warn/error), LOG_LEVEL env var,
    structured JSON in prod, human-readable in dev, redact() helper
  - Request ID middleware: X-Request-Id header on every response,
    request_end log with method/url/status/duration
  - Process-level: uncaughtException (exit 1), unhandledRejection handlers
  - /api/health: pings DB, 200 vs 503
  - PII removed from all logs (candidateId/jobId only, no name/email/phone)
  - All console.log migrated to logger (CLI scripts intentionally kept on console)

SECTION 8 — Security Hardening
  - Startup env validation: JWT_SECRET, ANTHROPIC_API_KEY always required;
    DB_HOST, DB_PASSWORD required in production — crash-early on missing
  - JWT secret hardcoded fallback removed
  - Helmet: CSP, HSTS 1yr + preload, frameguard deny, strict referrer policy
  - HTTPS enforcement via x-forwarded-proto in production (AWS ALB compatible)
  - DB SSL: rejectUnauthorized=true by default
  - CORS: HTTP S3 origin removed from production allowlist
  - Root / endpoint no longer leaks environment name

  - .eslintrc.json: eslint:recommended + no-unused-vars, eqeqeq, no-var,

  ENVIRONMENT VARIABLES (backend-node/.env)
================================================================================

  PORT                        — Express listen port (default 8080)
  NODE_ENV                    — 'production' or 'development'
  JWT_SECRET                  — REQUIRED: signs auth tokens
  ANTHROPIC_API_KEY           — REQUIRED: Claude API
  DB_HOST                     — REQUIRED in prod: RDS endpoint
  DB_PORT                     — default 5432
  DB_NAME                     — database name
  DB_USER                     — DB username
  DB_PASSWORD                 — REQUIRED in prod
  DB_SSL_REJECT_UNAUTHORIZED  — 'false' to disable SSL cert validation (RDS)
  AWS_ACCESS_KEY_ID           — S3 access
  AWS_SECRET_ACCESS_KEY       — S3 secret
  AWS_REGION                  — e.g. us-east-1
  AWS_S3_BUCKET               — resume storage bucket name
  GOOGLE_CLIENT_ID            — OAuth2 client ID
  GOOGLE_CLIENT_SECRET        — OAuth2 client secret
  GOOGLE_REDIRECT_URI         — OAuth2 callback URL
  LOG_LEVEL                   — debug/info/warn/error (default: info)
  EMAIL_FROM                  — SES sender address
  FRONTEND_URL                — for CORS allowlist

  See backend-node/.env.example for full documented list.
